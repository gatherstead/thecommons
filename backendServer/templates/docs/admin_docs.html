Access & Authentication

Login uses Django's built-in username/password. Any user with is_staff=True can log in. Superusers (is_superuser=True) have full access including user management.

python manage.py createsuperuser

Ingestion Section

Event Sources

The list of ICS calendar feeds the pipeline polls. To add a new calendar source, create a record here with the feed URL and set active = True.

FieldDescription
nameHuman-readable label (e.g. "Northampton Arts Calendar")
source_typeics, scraper, or email
urlThe ICS feed URL
activeUncheck to skip this source during ingestion
poll_interval_hoursHow often to re-poll (default: 24h)
last_polledRead-only; updated automatically after each poll
notesInternal notes

Bulk action: "Run ingestion pipeline (poll → standardize → dedup)" — triggers the full pipeline across all active sources.

Raw Events

Events exactly as they were parsed from ICS feeds — no editing or LLM processing. This is a permanent audit log of what was scraped. Use the processed filter to see which events are still waiting for standardization.

FieldDescription
raw_titleTitle exactly as scraped
raw_descriptionDescription exactly as scraped
raw_locationLocation string from the ICS feed
source_urlURL extracted from the event (used for webpage scraping)
source_uidICS UID or SHA256 hash — prevents re-importing
processedTrue once Gemini standardization has run on this event

Staged Events ← main review queue

Events cleaned by the LLM and waiting for admin review before going live. This is where most admin work happens.

Filter by status = pending to see your review queue.

StatusMeaning
pendingNeeds admin review
approvedPublished (or will be on next publish run)
rejectedWill not be published
duplicateAuto-flagged as near-duplicate of another staged event

How to review:

  1. Filter by pending
  2. Check title, description, town, date, price, tags, and the link
  3. Option A — Inline: click the Status dropdown in the list row and save
  4. Option B — Bulk action: check rows → "Approve selected events" or "Reject selected events" → Go
Approving creates the Event record immediately, attaches tags, and links the staged event to it. Rejecting just marks it as rejected — nothing is deleted.
FieldDescription
titleLLM-standardized title
descriptionLLM-written 2–3 sentence description
location_nameVenue or address
townTown inferred by LLM
tagsJSON list of tag strings
price0 = free, -1 = N/A, null = unknown
linkURL to the event's website
duplicate_ofFK to the earlier staged event it duplicates
reviewer_notesFree text for internal comments
raw_eventRead-only link back to the original scraped data

Events Section

Published Events

All events currently live on thecommons.town. These can be created, edited, or deleted directly here — changes take effect immediately on the website.

FieldDescription
titleEvent name
townTown/city
dateEvent datetime (UTC)
venueVenue name
descriptionFull description shown on the site
priceDecimal; null = unknown, 0 = free
photoOptional image upload
tagsMany-to-many relationship with Tag
linkURL to the event's page

Tags

Simple lookup table of tag strings (e.g. free, live-music, family-friendly). Shared across events and user profiles.

To add a new tag that the LLM can assign, create it here AND add it to the allowed tags list in ingestion/standardizer.py.

Users Section

Users

Standard Django user management. Use this to create staff accounts, reset passwords, and grant/revoke is_staff or is_superuser flags.

API Endpoints

Two backend endpoints exist outside the admin UI but are relevant to admin operations:

EndpointAuthWhat it does
GET /api/cron/ingest Bearer CRON_SECRET Triggers full ingestion pipeline. Called by Vercel cron at 8 AM UTC daily.
POST /api/events/publish-approved Bearer THE_COMMONS_API_KEY Publishes all approved staged events to the live Events table, then deletes them from staged. Atomic — rolls back on any failure.

Admin appearance is configured in backend/settings.py under the UNFOLD dict.

Custom admin page templates live in: backendServer/templates/docs/

Custom admin page routes must be declared before path("admin/", admin.site.urls) in backend/urls.py to avoid the admin catch-all swallowing them.
