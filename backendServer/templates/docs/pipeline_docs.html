Event Ingestion Pipeline

How ICS calendar events get from a source URL to a published event on thecommons.town.

Overview

ICS Source URL
     ↓  (Phase 1: Poll)
  RawEvent  (stored as-is)
     ↓  (Phase 2: Standardize)
 StagedEvent  (LLM-cleaned, status=pending)
     ↓  (Phase 3: Deduplicate)
 StagedEvent  (duplicates marked, status=duplicate)
     ↓  (Phase 4: Admin Review)  ← you are here
 StagedEvent  (status=approved or rejected)
     ↓  (Phase 5: Publish)
    Event  (live on website)
     ↓
  REST API  →  Frontend

Automated trigger: Vercel cron hits GET /api/cron/ingest every day at 8 AM UTC.

Manual trigger: Admin → EventSources → select source → "Run ingestion pipeline" action.

CLI: python manage.py ingest_events

Phase 1 — Poll ICS Sources

File: ingestion/importers/ics_importer.py · Model: RawEvent

  1. Load all EventSource records where active=True.
  2. Skip sources polled within the last poll_interval_hours (default: 24h).
  3. HTTP GET the ICS feed URL (30s timeout).
  4. Parse the .ics file with icalendar, extracting each VEVENT: SUMMARY, DESCRIPTION, LOCATION, DTSTART/DTEND, UID, and URL.
  5. Skip past events.
  6. Save as RawEvent with processed=False. Unique on (source, source_uid) — no re-imports.
  7. Update EventSource.last_polled = now().

Phase 2 — Standardize with LLM

File: ingestion/standardizer.py · Model: StagedEvent

  1. For each unprocessed RawEvent, fetch the event webpage (10s timeout) and extract visible text (max 6000 chars) using BeautifulSoup.
  2. Send raw event data + webpage text to Google Gemini (gemini-2.5-flash-lite → flash → pro fallback chain, 3 retries with backoff).
  3. Gemini returns: clean title, 2–3 sentence description, location name, town, tags (from allowed list), and price.
  4. Save as StagedEvent with status='pending'. Mark RawEvent.processed=True.

Phase 3 — Deduplicate

File: ingestion/deduplicator.py

  1. For each pending staged event, look for others within a ±3 hour window.
  2. Compare using thefuzz (Levenshtein distance): title similarity ≥ 80% and location similarity ≥ 75%.
  3. If both match, mark the newer one status='duplicate' with a duplicate_of FK.

Phase 4 — Admin Review

This page. See Staged Events →

  1. Filter by status=pending to see events awaiting review.
  2. Edit status inline in the list view, or select events and use the bulk action "Approve selected events".
  3. Approving immediately creates the Event record in the database (and attaches tags).
  4. "Reject selected events" marks them status=rejected — they won't be published.

Phase 5 — Publish to Website

File: ingestion/views.py → publish_approved_events()

  1. Find all StagedEvent records with status='approved' not yet in the Events table.
  2. Within an atomic transaction: create Event records + attach tags.
  3. Delete all approved staged events (cleanup).

Frontend Display

The frontend calls GET /events/ on load. eventService.ts transforms the response (formats date/time, formats price as "$X.XX" or "Free Entry"), then renders events as cards sorted by date. Users can filter by town and tags.

CLI Flags (for debugging)

python manage.py ingest_events                   # full pipeline
python manage.py ingest_events --skip-poll        # skip ICS fetch
python manage.py ingest_events --skip-standardize # skip Gemini step
python manage.py ingest_events --skip-dedup       # skip dedup
